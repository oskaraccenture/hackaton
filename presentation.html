The Patient — spring-music

What we inherited

  • Multi-DB profile coupling — Album.java uses only @Entity; three separate repo classes (JPA / MongoDB / Redis) all bundled in one JAR, activated by Spring profile at startup
  • Cloud Foundry lock-in — manifest.yml + java-cfenv-boot reads VCAP_SERVICES to activate DB profiles; infra baked into the app
  • Bundled frontend — AngularJS compiled into the JAR, no independent deployment
  • Zero API contract — no OpenAPI spec; consumers depend on observed behavior
  • No characterization tests — behavior is completely unpinned

The board's ask

  • Remove Cloud Foundry dependency
  • Enable independent service deployment
  • Explicit API contract for consumers
  • Testable — before and after every change

Stack

  • Legacy: Spring Boot 2.x, Gradle, H2/MySQL/MongoDB/Redis
  • New service: Node.js + Express, SQLite in-memory

The Strategy — Strangler Fig

Extract one service at a time. The monolith stays live. Each service is independently provable before any traffic shift.

Anti-patterns identified

  • Multi-DB profile coupling — all three repo classes (JPA, MongoDB, Redis) bundled in one JAR; SpringApplicationContextInitializer reads VCAP_SERVICES and activates one at startup
  • SPA bundled in JAR — AngularJS served as WebJars from the Spring Boot artifact; cannot deploy UI independently
  • CF lock-in — manifest.yml + java-cfenv-boot ties deployment to Cloud Foundry; infra baked into app
  • Zero API contract — no OpenAPI spec; consumers depend on observed HTTP behavior
  • No characterization tests — behavior completely unpinned before extraction

Extraction seams

  • Album CRUD REST boundary in AlbumController.java
  • Repository interfaces per DB: /jpa/, /mongodb/, /redis/

Improvements over legacy

  • Decoupled deploy — each container ships and scales independently
  • Clean API contract — no Spring annotations in public responses; ACL enforces boundary
  • Zero CF dependency — no manifest.yml, no java-cfenv-boot; deploy anywhere
  • Single DB per service — one storage technology, no bundled multi-DB repo selection

Key Decisions (ADRs)

ADR-001 — Strangler Fig

Monolith stays live. Extract one service at a time. Characterization tests pin behavior before any extraction.

ADR-002 — spring-music as legacy (option B)

Real code, real anti-patterns: all three DB repo classes (JPA / MongoDB / Redis) bundled in one JAR with profile-based activation via java-cfenv-boot, genuine CF coupling. More instructive than generated ugliness.

ADR-003 — PreToolUse hook vs. CLAUDE.md prompt

Hook = deterministic hard block for Spring annotations leaking into service routes.
Prompt = probabilistic preference for using the ACL adapter.
Rule: if failure mode is silent contract violation → hook. If suboptimal but correctable → prompt.
